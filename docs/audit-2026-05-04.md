# Audit projet — admin-dashboard
Date : 2026-05-04
Périmètre : architecture, code quality, performance, dette technique
Méthode : exploration code (46 pages, 93 routes API, 25 schemas, 26 services) + vérification ciblée DB/imports.

---

## Verdict global

Projet **mature et fonctionnel**, UX riche (6 thèmes, multi-tracks, responsive), bien organisé en services/schemas. **Trois axes prioritaires** ressortent :
1. **Cohérence des réponses API** (le frontend doit gérer N formats différents)
2. **Cache absent** (chaque clic = nouvelle requête DB)
3. **Auth hybride Stack+NextAuth/Authentik** : double système, défauts incohérents

Le reste est essentiellement du polissage : doublons, dead code, types `any` éparpillés.

---

## 🔴 Priorité 1 — Quick wins (impact fort, < 1h chacun)

### 1.1 Typo : `clien-import.tsx` (composant dupliqué)
- **Fichiers** : `components/clien-import.tsx` ET `components/client-import.tsx` (deux fichiers identiques)
- **Importé via le typo** : `app/(dashboard)/students/page.tsx:12`
- **Impact** : maintenance dangereuse, on peut éditer un fichier et oublier l'autre.
- **Fix** : supprimer `clien-import.tsx`, corriger l'import vers `client-import.tsx`.

### 1.2 Format de réponse API non standardisé
- **Symptôme** : selon la route, on reçoit `{success, promotions}`, `{success, data}`, `{success, promos}`, `{success, rows}`, ou un array brut.
- **Coût client** : chaque consommateur fait du pattern-matching ad hoc.
- **Fix** : adopter `{ success: bool, data?: T, error?: { code, message } }` pour TOUTES les routes. Helper `apiResponse()` / `apiError()` à créer dans `lib/api/response.ts`. Migrer route par route (ou via codemod).

### 1.3 Auth check répété dans chaque route
- **Pattern observé** : `const user = await stackServerApp.getUser(); if (!user) return NextResponse.json({error: 'Non authentifié'}, {status: 401});` répliqué partout.
- **Fix** : créer `lib/api/with-auth.ts` :
  ```ts
  export function withAuth<T>(handler: (req, user) => Promise<T>) {
    return async (req) => {
      const user = await stackServerApp.getUser();
      if (!user) return NextResponse.json({success:false, error:{code:'UNAUTHENTICATED'}}, {status:401});
      return handler(req, user);
    };
  }
  ```
- **Bonus** : combiner avec `withErrorHandler()` pour try/catch centralisé.

### 1.4 Logs en production
- **Volumétrie** : 205 occurrences de `console.*` dans le code prod (incluant des debug commentés dans `middleware.ts` lignes 51-225).
- **Fix** : créer `lib/log.ts` minimal (3 niveaux : `debug`/`info`/`error`) qui ne sort que si `NODE_ENV !== 'production'` pour `debug`/`info`. Remplacer progressivement. Supprimer les blocs commentés dans `middleware.ts`.

### 1.5 Pages déjà supprimées : nettoyer les liens orphelins
- Vérifier qu'il ne reste plus de `Link href="/reports"` ou `/promos/status` ou `/non-admin` nulle part (déjà fait dans les phases précédentes, mais utile à re-checker via grep avant un release).

## 🟠 Priorité 2 — Architecture (effort moyen, gain net)

### 2.1 Stratégie d'auth : choisir entre Stack Auth et NextAuth/Authentik
- **État** : double système actif. Stack Auth + NextAuth coexistent dans middleware + layout + nav-user.
- **Symptômes** : double route login (`/login` + `/auth/signin`), `planningPermission` hardcodé à `'reader'` côté Authentik alors que Stack Auth porte la vraie valeur, `ensure-user-metadata.ts` est un patch pour webhook incomplet.
- **Recommandation** : choisir l'un, migrer les comptes manquants côté retenu, supprimer l'autre. Stack Auth semble plus aligné avec votre stack (typed metadata, hooks React, gestion fine des roles).
- **Effort** : ~1 semaine si tout le monde migre Stack, plus si vous gardez Authentik.

### 2.2 Cache : aucune stratégie active
- **Constat** : zéro usage de `revalidate`, `unstable_cache`, `cache`. Chaque montage de composant client = nouveau fetch.
- **Quick wins** à mettre en place :
  - `/api/promotions`, `/api/projects`, `/api/holidays` : tags `next: { revalidate: 3600 }` (config peu changeante)
  - `/api/widgets/overview` : `unstable_cache(fn, ['overview'], { revalidate: 300 })`
  - `/api/code-reviews/stats` : agrégations lourdes → `revalidate: 600`
- **Bonus** : envisager `swr` ou `react-query` pour les composants client (déjà déduplication des fetches, optimistic UI).

### 2.3 Helpers de loader config redondants
- **Fichiers** : `lib/config/{holidays,projects,promotions}.ts` exposent tous le même pattern `dbXyzToConfig()` + service wrapper.
- **Fix** : extraire un `lib/config/loader.ts` générique `dbRowsToConfig<TRow, TConfig>(rows, mapper)`. Économie : ~50% de code.

### 2.4 Services thin (1-line wrappers)
- **Exemples** : `holidays.ts`, `discordUsers.ts`, parties de `employees.ts` ne sont que des passe-plats Drizzle.
- **Recommandation** : soit les enrichir (validation, transformation, cache), soit les inliner. Le cas par cas. Pas de règle "toujours service" → privilégier la lisibilité.

### 2.5 N+1 dans les crons
- `app/api/cron/notify-audit-groups/route.ts` : 3 boucles imbriquées (groupes → membres → notifs) + appels Discord/email sériels.
- **Fix** :
  - Batch les requêtes DB (un seul `IN (...)` au lieu de loops)
  - `Promise.all` pour les appels Discord (déjà fait sur `apply-rotation`, à généraliser)
  - Ajouter un lock distribué (cookie + timestamp dans `cron_runs` table) pour éviter doubles envois.

### 2.6 Intégrations externes sans timeout / retry
- **Risque** : Gitea, Mistral, Discord → si l'API est lente, votre app hang sans feedback.
- **Fix** : wrapper `fetch` partagé dans `lib/services/http.ts` avec `AbortController` (timeout 8s) + 1 retry sur erreur transitoire.

## 🟡 Priorité 3 — Code quality (long terme)

### 3.1 Types `any` (~40 occurrences)
- Lieux : `NavUser` props, form render functions, MDX components.
- **Fix** : un `any` à la fois, remplacer par un type précis. Quick win immédiat : `NavUser` peut typer `user` proprement (déjà un type `User` défini).

### 3.2 Type duplications
- `User`, `PromoConfig`, `TimeSlot` sont souvent redéclarés inline dans des composants au lieu d'importer depuis `lib/db/schema/*`.
- **Fix** : un grep régulier `interface User\|type User\|interface Promo\b` puis remplacer par les types canoniques.

### 3.3 Fonctions services orphelines
- `lib/db/services/projects.ts` : `getPromoById`, `getHolidayById`, `getProjectsByCategory`, `getProjectById` → aucun import.
- **Fix** : supprimer.

### 3.4 `eventIdToString()` orphelin dans `lib/config/promotions.ts`
- Wrapper trivial `String(eventId)` jamais utilisé. Supprimer.

### 3.5 i18n : 100 % français hardcodé
- Pas de framework i18n (`next-intl`, `i18next`). Toutes les chaînes UI sont inline.
- **Risque** : si un jour vous voulez de l'anglais, c'est un chantier sur 46 pages + 93 routes.
- **Recommandation** : non urgent. Évaluer le besoin réel avant d'investir.

### 3.6 Tests
- Pas vu de fichiers `*.test.ts` ni `*.spec.ts`. Aucune CI test step.
- **Recommandation** : commencer par les helpers purs (`lib/timeline.ts`, `lib/db/utils.ts`, `nav-apps.ts` `findActiveApp`). Vitest est rapide à mettre en place.

## 🟢 Priorité 4 — Petites améliorations UX

### 4.1 `/login` vs `/auth/signin`
- Deux pages de login coexistent (Stack Auth + NextAuth). Source de confusion utilisateur.
- **Fix** : rediriger `/auth/signin` vers `/login` (ou inverse). Idéalement résolu par la consolidation auth (2.1).

### 4.2 Pages mobile
- Sauf le planning et le dashboard étudiant, peu de pages ont une expérience mobile dédiée. Tailwind breakpoints couvrent les cas faciles mais certains tableaux longs (`/students-table`) sont compliqués à scroller.
- **Recommandation** : audit page-par-page à faire après les Priorités 1/2.

### 4.3 Tâches du Hub
- Schéma `hub_*` en place mais peu d'enregistrements en DB (1 template, 3 tasks). Soit on finalise la refonte, soit on archive le module.

## Verdict

Le projet est **dans une bonne forme générale**. La complexité accumulée est typique d'une app qui a évolué rapidement avec plusieurs vagues de features (auth hybride, refonte sidebar, dashboard étudiant, mobile). Les 3 chantiers principaux (réponses API, cache, auth) représentent ~2 semaines de travail focus et apporteraient une **base bien plus solide** pour la suite. Le reste est du polissage qui peut se faire au fil de l'eau.

Aucune dette technique critique au sens « bloquante demain ». Aucune faille de sécurité évidente côté auth (les guards admin sont en place). Mobile fonctionne, thèmes fonctionnent, tâches CR/planning/specialty sont opérationnelles.

**Recommandation** : faire les **5 quick wins du Sprint 1 cette semaine** (effort total ~6h), puis budgéter le Sprint 2 (cache) dans les 2 prochaines semaines. Le Sprint 3 (auth) peut attendre que le besoin se fasse sentir.
